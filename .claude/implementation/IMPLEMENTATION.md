# Implementation Tracking -- The Accountability Layer

The most dangerous gap in entrepreneurial decision-making is not the gap between ignorance and knowledge. It is the gap between knowing and doing. A founder who completes Porter's Five Forces analysis and files it away has gained nothing. A founder who runs the pivot playbook, produces a brilliant action plan, and then returns to the same habits three weeks later has consumed effort without creating change. Foreman's diagnostics identify problems. Its skills produce frameworks. Its playbooks chain frameworks into comprehensive strategies. Implementation tracking ensures those strategies leave the document and enter the business.

This system transforms Foreman from an advisor that produces recommendations into a partner that tracks whether recommendations become reality.

---

## What Implementation Tracking Is

Implementation tracking is the persistence and accountability layer that monitors the execution of recommendations produced by skills, diagnostics, and playbooks. It is not project management. It does not replace Jira, Asana, or a whiteboard. It tracks the specific outputs of Foreman's analytical system -- the five-step action plan from a playbook, the three strategic priorities from a SWOT analysis, the pricing change recommended by a diagnostic -- and holds the entrepreneur accountable for acting on them.

Every recommendation that emerges from the system becomes a trackable implementation item. Each item carries a status, a progress percentage, a deadline, an owner, and a history of check-ins. When items stall, the system detects it. When blockers emerge, the system categorizes them and routes to the appropriate diagnostic or skill. When items complete, the system records the outcome and moves them to the history layer.

The operating assumption: producing insight is 20% of the value. Executing on insight is the remaining 80%. This system exists for the 80%.

---

## How It Works

### Activation

Implementation tracking activates automatically in three scenarios:

1. **Playbook step completion.** When the playbook-runner-agent completes a step that produces actionable recommendations, each recommendation becomes an implementation item.
2. **Skill application with action items.** When the skill-executor-agent produces output containing explicit recommendations (not all skill outputs are actionable -- a SWOT analysis is informational, but "restructure pricing tier" is actionable), those recommendations are extracted as implementation items.
3. **Diagnostic routing.** When a diagnostic produces a root-cause finding with a prescribed next step, that next step becomes an implementation item.

### Research-Originated Items

Research tasks generated by `/research` commands or recommended by agents are automatically registered as implementation items with source type `research-guide`. They track research progress through the same status workflow (`not-started` --> `in-progress` --> `completed`). When a research item completes, the system prompts: "Research complete. Ready to apply the skill that needs this data?" The linked skill is derived from the research guide's `feeds_into_skills` frontmatter.

### Item Creation

Each implementation item is created with the following properties:

- **id** -- Unique identifier (e.g., `impl-001`)
- **source** -- Where the item originated: playbook step, skill recommendation, or diagnostic routing
- **source_ref** -- Specific reference to the originating step or skill
- **description** -- What needs to be done, in concrete terms
- **owner** -- Who is responsible (default: "self" for solopreneurs and founders without delegation)
- **created** -- Date the item was created
- **deadline** -- Target completion date
- **status** -- Current state (see Implementation States below)
- **progress** -- Percentage complete (0-100)
- **blockers** -- Active blockers preventing progress
- **notes** -- Running notes and observations
- **check_ins** -- History of progress updates from weekly check-ins

### Weekly Check-In Cycle

Every seven days, the system prompts for a progress review via the `/check-in` command. The check-in walks through all active implementation items, records updates, detects stalls, and flags blockers. This cadence is not arbitrary -- it matches the active memory layer's update frequency and prevents items from drifting into neglect without creating daily overhead.

### Stall Detection

When an implementation item remains at the same progress percentage for two or more consecutive check-ins (14+ days without movement), the system flags it as stalled. Stalled items trigger a blocker analysis: the system asks targeted questions to identify why progress has stopped and categorizes the blocker for routing.

## Implementation States

| State | Meaning | Transitions To |
|-------|---------|----------------|
| `not-started` | Recommendation produced, not yet acted on | `in-progress`, `abandoned`, `deferred` |
| `in-progress` | Work has begun, progress tracked as percentage | `completed`, `blocked`, `abandoned`, `deferred` |
| `blocked` | Explicitly blocked, blocker identified and categorized | `in-progress` (when blocker resolves), `abandoned`, `deferred` |
| `completed` | Successfully implemented, outcome recorded | -- (terminal state, moves to history) |
| `abandoned` | Deliberately stopped, reason recorded | -- (terminal state, moves to history) |
| `deferred` | Postponed to a specific future date with justification | `not-started` (when defer date arrives) |

State transitions are recorded with timestamps. The history of state changes provides a behavioral signal: a founder who frequently moves items from `in-progress` to `deferred` may be avoiding execution. A founder whose items cluster in `blocked` may need different support than one whose items cluster in `not-started`.

---

## Blocker Categories

When an item enters `blocked` status, the blocker is classified into one of five categories. Classification drives routing -- each category maps to specific diagnostics, skills, or system responses.

| Category | Description | Routing |
|----------|-------------|---------|
| **Resource** | Insufficient time, money, or people to execute | Route to `resource-allocation` skill, `operations-diagnosis` |
| **Knowledge** | Do not know how to execute the recommendation | Route to relevant skill for deeper application, or surface related skills |
| **Dependency** | Waiting on an external party, deliverable, or decision | Route to `stakeholder-map` template, `partnership-diagnosis` if applicable |
| **Motivation** | Avoiding the task -- psychological resistance, fear of failure, discomfort | Route to `stoic-emotional-mastery`, `stoic-control-and-clarity`, `founder-mental-health-check` |
| **External** | Market conditions, regulatory requirements, or partner constraints beyond control | Route to `scenario-planning` skill, `swot-pest-analysis` for reassessment |

Motivation blockers deserve special attention. They are the most common and the least reported. The system surfaces them by inference: when a founder cannot articulate a resource, knowledge, dependency, or external blocker, the system gently probes whether the true obstacle is internal resistance.

## Check-In Protocol

The `/check-in` command initiates the weekly accountability review. The protocol proceeds as follows:

**Phase 1 -- Status Sweep.** The system presents all active implementation items (status: `not-started`, `in-progress`, `blocked`) with their current progress and last check-in date. Items are sorted by deadline proximity.

**Phase 2 -- Progress Update.** For each item, the system asks: "What progress have you made on [description] since [last check-in date]?" The entrepreneur provides an updated percentage and a brief note. If the percentage has not changed, the system probes for blockers.

**Phase 3 -- Blocker Identification.** For any item with zero progress delta, the system asks: "What is preventing progress?" Responses are categorized into the five blocker types. If the entrepreneur cannot identify a specific blocker, the system asks: "Is this something you have been avoiding? If so, what makes it uncomfortable?" This surfaces motivation blockers without accusation.

**Phase 4 -- Priority Rebalancing.** After all items are reviewed, the system identifies the three highest-priority items for the coming week based on deadline proximity, strategic importance (source playbook or skill), and blocker resolution potential. These become the recommended focus.

**Phase 5 -- Memory Persistence.** All updates are written to the active memory layer. Completed items move to history. Check-in data is appended to each item's check_in array. The session layer records the check-in event.

## Integration with Memory

Implementation tracking extends two existing memory layers without replacing them.

**Active layer extension.** The active layer gains an `implementation_items` array alongside `current_playbook`, `pending_decisions`, and `follow_ups`. This array holds all items in non-terminal states. The memory-agent reads and writes this array during session start and end.

**History layer extension.** The `decisions_made` section in the history layer gains implementation tracking fields: `implementation_status`, `implementation_started`, `implementation_completed`, and `implementation_notes`. When an implementation item reaches a terminal state (`completed` or `abandoned`), its data enriches the corresponding history entry. The `playbooks_completed` section gains similar fields to track which playbook recommendations were actually executed versus merely produced.

## Integration with Diagnostics

Blocker detection creates a feedback loop into the diagnostic system. When a blocker is identified and categorized, the system checks whether a relevant diagnostic exists:

- Resource blocker with cash implications --> `fundraising-diagnosis` or `operations-diagnosis`
- Knowledge blocker in a specific domain --> route to the skill that produced the original recommendation for deeper application
- Dependency blocker involving a partner --> `partnership-diagnosis`
- Motivation blocker with burnout signals --> `leadership-diagnosis` with `founder-mental-health-check` template
- External blocker requiring strategic reassessment --> `strategy-diagnosis` or `competition-diagnosis`

This routing is automatic. The system does not merely flag the blocker -- it offers the next step.

## Integration with Stoic Mode

When `/stoic on` is active, the implementation tracking system gains philosophical depth. Check-ins include Stoic reflection prompts drawn from the twelve Stoic skills:

- **Dichotomy of control.** For each blocker: "Is this within your control, within your influence, or entirely external? Are you spending energy on what you cannot change?"
- **Premeditatio malorum.** At each check-in: "What could go wrong with your highest-priority item this week? Have you prepared for it, or are you assuming a clear path?"
- **Amor fati.** For abandoned items: "Can you embrace this outcome rather than resent it? What does this abandonment teach you about your actual priorities versus your stated ones?"
- **Motivation blockers** receive specific Stoic framing: "Which of the Seven Functions of the Mind is failing here? Is this a Refusal problem (inability to say no to distractions), a Purpose problem (unclear why this matters), or a Preparation problem (you have not readied yourself for the difficulty)?"

Stoic mode does not soften the accountability. It deepens it.

---

## Integration with Solo Mode

When `/solo` is active, implementation tracking adapts for the solopreneur context:

- **Owner field defaults to "self" permanently.** No delegation tracking -- the solopreneur is every department.
- **Isolation-specific blocker detection.** The system adds a sixth blocker probe: "Are you blocked because you need help but have no team to ask? Would a contractor, advisor, or community solve this?" This surfaces the solopreneur's unique constraint -- the absence of colleagues to unblock them.
- **Capacity-aware prioritization.** During check-in Phase 4, the system factors in solopreneur capacity: "You have 12 active implementation items. A solo operator can realistically advance 3-5 per week. Which three matter most?"
- **Burnout detection.** If progress across all items drops simultaneously over two consecutive check-ins, the system flags potential burnout rather than assuming laziness: "Progress has stalled across the board. Before we discuss individual items -- how are you?"

Solo mode recognizes that the solopreneur's greatest implementation risk is not bad strategy. It is exhaustion.

## System Boundaries

Implementation tracking deliberately does not:

- **Replace external project management tools.** It tracks Foreman-originated recommendations, not the entrepreneur's full task list.
- **Assign arbitrary deadlines.** Deadlines are set by the entrepreneur or derived from playbook timelines. The system never invents urgency.
- **Judge the entrepreneur.** Stall detection and blocker analysis are diagnostic, not punitive. The system surfaces patterns. The entrepreneur decides what to do about them.
- **Track indefinitely.** Items that remain in `not-started` for 30 days are flagged for a decision: begin, defer with a date, or abandon with a reason. Zombie items are not permitted.

The goal is accountability with respect. A good mentor remembers what you committed to, asks how it went, and helps when you are stuck. That is the operating model.
